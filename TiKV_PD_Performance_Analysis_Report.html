📚 理论基础: Sujin Park 优化框架

本报告基于 OSDI 2025 论文 "Principles and Methodologies for Serial Performance Optimization" (Sujin Park, Mingyu Guan, Xiang Cheng, Taesoo Kim, Georgia Tech) 的系统性优化框架。

🎯 核心思想: 性能优化是科学，不是魔法。通过结构化框架识别瓶颈，应用对应方法论，系统性提升性能。

三大优化原则

原则 核心思想 应用场景
Task Removal 消除不必要的工作 识别并移除冗余计算、IO、RPC
Task Replacement 用更高效的实现替换 算法优化、架构重构
Task Reordering 调整执行顺序减少依赖 异步化、流水线优化

八种可操作方法论

Batching — 合并多个小操作 → 减少 syscall、RPC 次数
Caching — 复用先前计算结果 → 避免重复计算
Precomputing — 提前计算未来需要的结果 → 消除运行时延迟
Deferring — 延迟非紧急任务 → 解耦关键路径
Relaxation — 放宽一致性/精确性要求 → 减少同步等待
Contextualization — 利用上下文信息优化 → 位置感知、负载感知
Hardware Specialization — 针对特定硬件优化 → 异构计算、微服务拆分
Layering — 分层抽象降低开销 → 存储分层、职责分离

🏗️ PD 架构与核心性能瓶颈

TiKV PD (Placement Driver) 是 TiDB 集群的元数据管理和调度中心，承担以下核心职责:

模块 功能 潜在瓶颈 严重程度
TSO Allocator 全局时间戳分配 单点瓶颈、网络延迟 高
Region Metadata Region 路由信息管理 内存占用、查询延迟 中
Scheduling 负载均衡、副本调度 计算密集、IO 密集 中
Heartbeat 处理 TiKV 心跳 高频写入、etcd 压力 高
etcd Store 元数据持久化 事务延迟、一致性开销 中

🔷 原则 1: Task Removal (任务移除)

💡 核心思想: 消除不必要的工作

识别系统中的冗余计算、重复请求、无效写入，将其彻底移除。

优化点 1.1: TSO 批量预取减少 RPC 调用

📝 问题描述

每个事务多次请求 TSO，产生大量 RPC 调用，网络开销巨大。

✅ 优化方案

TiKV 侧批量预取 TSO，本地缓存使用，大幅减少 RPC 次数。

🎯 对应方法论

Batching + Caching

// 优化示例: TiKV 侧 TSO 预取配置
[causal-ts]
alloc-ahead-buffer = "6s"        # 预分配 6 秒的 TSO
renew-batch-max-size = 65536     # 最大批量
renew-batch-min-size = 2048      # 最小批量

// 原理: 从 N 次 RPC → 1 次批量 RPC

优化点 1.2: 移除不必要的 etcd 写入

📝 问题描述

心跳数据全量写入 etcd，产生大量磁盘 IO，影响性能。

✅ 优化方案

区分易失性数据和持久化数据，易失性数据存内存，持久化数据存 etcd。

🎯 对应方法论

Layering — 分层存储策略

// 数据分层策略
易失性数据 (Region 实时状态) → 内存存储 (fast path)
持久化数据 (Region 拓扑)     → etcd 存储 (slow path)

🔷 原则 2: Task Replacement (任务替换)

💡 核心思想: 用更高效的实现替换

保留功能语义，用更优的算法、架构或数据结构替换原有实现。

优化点 2.1: 微服务架构解耦 (PD v8.0+)

📝 问题描述

PD 单体架构，TSO/调度/路由相互影响，一个模块故障影响全局。

✅ 优化方案

拆分为独立微服务，各模块独立部署、独立扩展。

🎯 对应方法论

Hardware Specialization — 针对不同负载特化部署

微服务 职责 优化收益
tso 专精时间戳分配 可独立扩展，降低延迟
scheduling 专精调度决策 故障隔离，升级无感知
pd-server 专注元数据路由 简化逻辑，提升稳定性

优化点 2.2: Local TSO 替代 Global TSO (跨数据中心场景)

📝 问题描述

跨 DC 场景下，所有 TSO 请求都路由到 PD leader，延迟高达 20-50ms。

✅ 优化方案

引入 Local TSO allocator，本地事务直接本地授时，只有跨 DC 事务才需要 Global TSO。

🎯 对应方法论

Contextualization — 利用地理位置上下文优化

// Local TSO 架构
DC-1: Local TSO Allocator ← 本地事务 (低延迟 < 5ms)
DC-2: Local TSO Allocator ← 本地事务 (低延迟 < 5ms)
       ↓
Global TSO Allocator ← 跨 DC 事务 (同步开销)

🔷 原则 3: Task Reordering (任务重排序)

💡 核心思想: 调整执行顺序减少依赖/延迟

将阻塞操作改为异步，重排任务执行顺序，解耦关键路径。

优化点 3.1: 异步心跳处理

📝 问题描述

同步处理心跳阻塞关键路径，TiKV 等待响应超时风险。

✅ 优化方案

接收心跳后立即响应，异步处理状态更新。

🎯 对应方法论

Deferring — 延迟非紧急任务

// 原顺序: 同步处理
接收心跳 → 更新状态 → 持久化存储 → 响应心跳
   ↑___________________________________↓
              (阻塞 5-10ms)

// 优化后: 异步处理
接收心跳 → 入队列 → 立即响应
             ↓
          异步处理状态更新

优化点 3.2: TSO 物理时间推进优化

📝 问题描述

TSO 物理时间每 50ms 检查一次，短事务可能等待不必要的时间。

✅ 优化方案

动态调整检查间隔，高频更新物理时间戳。

🎯 对应方法论

Relaxation — 放宽严格性换取性能

# PD 配置优化
tso-update-physical-interval = "1ms"  # 从 50ms 优化到 1ms

4.1 TSO 分配路径优化

// pkg/tso/allocator.go
// 当前: 单批次分配
func (a *Allocator) GenerateTSO(count uint32) (pdpb.Timestamp, error) {
    // ... 同步分配逻辑，每次请求都走网络
}

// 优化: 批量预取 + 本地缓存
// 方法论: Batching + Caching

type TSOCache struct {
    buffer []pdpb.Timestamp
    index  int
    mu     sync.Mutex
}

func (c *TSOCache) Get(count uint32) (pdpb.Timestamp, error) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    if c.index+int(count) >= len(c.buffer) {
        // Batch fetching - Batching methodology
        c.refill()  // 一次性获取大批量 TSO
    }
    
    // Return from cache - Caching methodology
    ts := c.buffer[c.index]
    c.index += int(count)
    return ts, nil
}

4.2 Region 心跳处理优化

// server/cluster/cluster.go
// 当前: 同步更新
func (c *RaftCluster) processRegionHeartbeat(region *core.RegionInfo) error {
    // ... 直接更新存储，阻塞关键路径
    return c.storage.SaveRegion(region)
}

// 优化: 异步批处理
// 方法论: Batching + Deferring

func (c *RaftCluster) processRegionHeartbeat(region *core.RegionInfo) error {
    // 1. 立即更新内存状态 (关键路径，必须快)
    c.basicCluster.PreCheckPutRegion(region)
    
    // 2. 异步批量持久化 (非关键路径，延迟处理)
    c.heartbeatQueue.Push(region)
    
    return nil  // 立即返回，不阻塞
}

4.3 调度决策优化

// server/schedule/scheduler.go
// 优化: 预计算热点
// 方法论: Precomputing + Contextualization

type HotSpotPredictor struct {
    history map[uint64][]*RegionStat
    model   *MLModel  // 可接入 ML 预测模型
}

func (p *HotSpotPredictor) Predict(storeID uint64) *HotSpot {
    // 基于历史数据预计算热点
    // Contextualization: 利用历史上下文和趋势
    stats := p.history[storeID]
    return p.model.Predict(stats)
}

// 提前调度，而非被动响应
func (s *Scheduler) proactiveBalance() {
    for _, store := range s.stores {
        if hotspot := s.predictor.Predict(store.ID); hotspot != nil {
            s.preMoveRegion(hotspot.region, hotspot.targetStore)
        }
    }
}
