🔷 Principle 1: Task Removal

💡 Core Idea: Eliminate Unnecessary Work

Identify and remove redundant computation, duplicate parsing, and unnecessary data movement.

Optimization 1.1: Prepared Plan Cache

📝 Problem

Repeated SQL parsing and plan generation consume significant CPU for similar queries.

✅ Solution

Cache execution plans and reuse them for identical SQL statements.

🎯 Methodology

Caching

-- Enable Prepared Plan Cache
SET GLOBAL tidb_enable_prepared_plan_cache = 1;
SET GLOBAL tidb_prepared_plan_cache_size = 1000;

-- Go code integration
// session/session.go
func (s *session) ExecuteStmt(ctx context.Context, stmt ast.StmtNode) (sqlexec.RecordSet, error) {
    // Check Plan Cache first
    if plan, ok := s.planCache.Get(stmt); ok {
        return s.runPlan(ctx, plan) // Reuse cached plan
    }
    
    // Generate new plan if cache miss
    plan := optimizer.Optimize(ctx, stmt)
    s.planCache.Put(stmt, plan) // Cache for future use
    return s.runPlan(ctx, plan)
}

Optimization 1.2: Coprocessor Pushdown

📝 Problem

Transferring large amounts of raw data from TiKV to TiDB for processing creates network bottlenecks.

✅ Solution

Push computation down to TiKV nodes, only return aggregated results.

🎯 Methodology

Task Removal — Remove unnecessary data transfer

-- Query execution flow optimization
-- Before: TiKV returns 1M rows → TiDB calculates COUNT
-- After: TiKV calculates COUNT locally → returns 1 number

SELECT COUNT(*) FROM orders WHERE amount > 1000;

-- Coprocessor executes: TableScan + Selection + Aggregation
-- Network transfer: 8 bytes (result) vs 50MB (raw data)

Optimization 1.3: One-Phase Commit (1PC)

📝 Problem

Two-Phase Commit (2PC) adds latency for single-row transactions.

✅ Solution

Skip Prepare phase for single-Region transactions.

🎯 Methodology

Task Removal — Eliminate unnecessary coordination

// store/tikv/2pc.go
func (c *twoPhaseCommitter) execute(ctx context.Context) error {
    // Optimization: Detect single-Region, single-row transactions
    if c.isSingleRegion() && c.isSingleRow() {
        // Skip 2PC Prepare phase, commit directly
        return c.commitOnePhase(ctx)
    }
    // Fall back to standard 2PC for distributed transactions
    return c.commitTwoPhase(ctx)
}

🔷 Principle 2: Task Replacement

💡 Core Idea: Replace with More Efficient Implementation

Maintain functional semantics while using better algorithms, data structures, or architectures.

Optimization 2.1: Cost-Based Optimizer (CBO)

📝 Problem

Rule-based optimizer cannot adapt to changing data distributions.

✅ Solution

Use statistics to estimate costs and choose optimal execution plans.

🎯 Methodology

Contextualization — Leverage data distribution context

-- Collect statistics for CBO
ANALYZE TABLE orders;

-- Detailed statistics with histograms
ANALYZE TABLE orders INDEX idx_amount WITH NUM BUCKETS 256;

-- The optimizer uses these statistics to:
-- 1. Choose between index scan vs table scan
-- 2. Determine optimal join order
-- 3. Select appropriate join algorithms

Optimization 2.2: TiFlash Columnar Storage

📝 Problem

OLAP queries on row-oriented storage (TiKV) are inefficient for analytical workloads.

✅ Solution

TiFlash provides columnar storage with vectorized execution for HTAP.

🎯 Methodology

Hardware Specialization — Specialize storage for workload type

Workload Type Storage Engine Optimization
OLTP (Point queries, small range scans) TiKV (Row-oriented) Fast single-row access
OLAP (Aggregations, full table scans) TiFlash (Column-oriented) Vectorized execution, high compression
Hybrid (HTAP) TiKV + TiFlash with MPP Parallel execution across nodes

🔷 Principle 3: Task Reordering

💡 Core Idea: Adjust Execution Order to Reduce Dependencies

Convert blocking operations to asynchronous, pipeline execution to maximize parallelism.

Optimization 3.1: Async Commit

📝 Problem

Synchronous 2PC commit waits for all nodes, increasing latency.

✅ Solution

Return success after prewrite, commit asynchronously in background.

🎯 Methodology

Relaxation — Trade immediate durability for lower latency

-- Enable Async Commit
SET GLOBAL tidb_enable_async_commit = 1;

-- Transaction flow comparison
Traditional 2PC:  Prewrite → Commit → Wait all responses → Return (20ms)
Async Commit:     Prewrite → Local confirm → Async commit → Return (5ms)
                                        ↓
                                   Background commit

Optimization 3.2: Pipeline Execution

📝 Problem

Operators execute sequentially, underutilizing resources.

✅ Solution

Pipeline parallel execution with multiple operators running concurrently.

🎯 Methodology

Batching + Hardware Specialization

-- MPP (Massively Parallel Processing) execution
SELECT region, SUM(amount) 
FROM orders 
GROUP BY region;

-- Execution plan with MPP:
-- 1. Each TiFlash node scans local data
-- 2. Local partial aggregation (parallel)
-- 3. Exchange data for final aggregation
-- 4. Return results

Optimization 3.3: Async DDL

📝 Problem

Schema changes block table operations.

✅ Solution

Online DDL execution in background with multiple states.

🎯 Methodology

Deferring — Non-blocking schema changes
