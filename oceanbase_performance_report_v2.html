📚 理论基础: Sujin Park 优化框架

本报告基于 OSDI 2025 论文 "Principles and Methodologies for Serial Performance Optimization" (Sujin Park, Mingyu Guan, Xiang Cheng, Taesoo Kim, Georgia Tech) 的系统性优化框架。

🎯 核心思想:

性能优化是科学，不是魔法。通过结构化框架识别瓶颈，应用对应方法论，系统性提升性能。

三大优化原则

原则 核心思想 应用场景
Task Removal 消除不必要的工作 识别并移除冗余计算、IO、网络传输
Task Replacement 用更高效的实现替换 算法优化、架构重构、硬件加速
Task Reordering 调整执行顺序减少依赖 异步化、流水线优化、延迟执行

八种可操作方法论

Batching

合并多个小操作 → 减少 syscall、RPC 次数

Caching

复用先前计算结果 → 避免重复计算

Precomputing

提前计算未来需要的结果 → 消除运行时延迟

Deferring

延迟非紧急任务 → 解耦关键路径

Relaxation

放宽一致性/精确性要求 → 减少同步等待

Contextualization

利用上下文信息优化 → 负载感知、位置感知

Hardware Specialization

针对特定硬件优化 → 异构计算、RDMA、GPU

Layering

分层抽象降低开销 → 存储分层、职责分离

🔷 原则 1: Task Removal (任务移除)

💡 核心思想: 消除不必要的工作

识别系统中的冗余计算、重复请求、无效写入，将其彻底移除。

优化点 1.1: 执行计划去重缓存

📝 问题描述: 参数化查询每次生成新计划，即使查询模式完全相同，也重复进行语法解析和代价估算。

✅ 优化方案: 基于查询模板缓存执行计划，相同模式的查询直接复用缓存计划。

🎯 对应方法论: Caching — 复用先前计算结果

// 优化配置: 执行计划缓存参数
set global ob_plan_cache_percentage = 20;        # 缓存内存占比
set global ob_plan_cache_evict_high_percentage = 90;  # 高水位线

// SQL Hint 使用缓存
SELECT /*+ USE_PLAN_CACHE(none) */ ...;          # 强制不缓存
SELECT /*+ USE_PLAN_CACHE(default) */ ...;       # 默认缓存策略

优化点 1.2: 消除重复的热点数据解压缩

📝 问题描述: 热点行频繁读取时，每次都要从 SSTable 解压缩，造成大量重复计算。

✅ 优化方案: 引入行级缓存层，热点数据常驻内存，避免重复解压缩。

🎯 对应方法论: Caching + Layering — 分层缓存策略

// 行缓存配置
alter system set enable_row_cache = true;
alter system set row_cache_priority = 'HIGH';    # 热点行优先缓存

// 查询时指定缓存策略
SELECT /*+ READ_CONSISTENCY(WEAK) */ * FROM hot_table WHERE id = 1;
// 弱一致读可命中行缓存，避免 SSTable 访问

优化点 1.3: 数据-日志分离存储

📝 问题描述: 传统 Paxos 要求数据和日志同副本数存储，存储成本高。

✅ 优化方案: 日志 5 副本保证高可用，用户数据 2-4 副本，移除冗余数据副本。

🎯 对应方法论: Layering — 存储分层，职责分离

// 数据-日志分离配置
ALTER SYSTEM SET data_copy_count = 2;      # 用户数据 2 副本
ALTER SYSTEM SET log_copy_count = 5;       # 日志 5 副本

// 收益: 存储成本降低 20-40%，同时保持相同可用性

🔷 原则 2: Task Replacement (任务替换)

💡 核心思想: 用更高效的实现替换

保留功能语义，用更优的算法、架构或数据结构替换原有实现。

优化点 2.1: 自适应执行引擎 (Context-aware)

📝 问题描述: 固定执行策略无法适应 OLTP/OLAP 混合负载波动，资源竞争严重。

✅ 优化方案: 根据负载上下文动态选择执行策略，OLTP 用行存，OLAP 用列存。

🎯 对应方法论: Contextualization + Layering — 负载感知 + 存储分层

// 自适应执行策略选择
class AdaptiveExecutionEngine {
    ExecutionPolicy SelectPolicy(WorkloadContext ctx) {
        if (ctx.type == OLTP && ctx.cpu_usage < 70%) {
            // OLTP 低负载: 激进缓存策略
            return AggressiveCachePolicy();
        }
        if (ctx.type == OLAP || ctx.io_pressure > 80%) {
            // 高 I/O 压力: 启用列存扫描
            return ColumnarScanPolicy();
        }
        if (ctx.memory_pressure > 90%) {
            // 内存紧张: 减少缓存，增加 Spill
            return MemoryConstrainedPolicy();
        }
        return DefaultPolicy();
    }
};

🔷 原则 3: Task Reordering (任务重排序)

💡 核心思想: 调整执行顺序减少依赖/延迟

将阻塞操作改为异步，重排任务执行顺序，解耦关键路径。

优化点 3.1: 异步 Compaction 调度

📝 问题描述: 同步 Compaction 阻塞写入，高峰期影响事务延迟。

✅ 优化方案: 高峰期延迟 Compaction，低峰期批量执行，解耦关键路径。

🎯 对应方法论: Deferring — 延迟非紧急任务

// 原顺序: 同步处理 (阻塞)
写入数据 → MemTable 满 → 立即 Flush → SSTable → 响应客户端
                    ↑________________________________________↓
                              (阻塞 5-10ms)

// 优化后: 异步处理 (非阻塞)
写入数据 → MemTable 满 → 入队列 → 立即响应
                        ↓
                     后台异步 Flush (低峰期执行)

// 配置参数
ALTER SYSTEM SET minor_merge_concurrency = 4;      // 限制并发
ALTER SYSTEM SET minor_freeze_times = 10;          // 触发阈值

优化点 3.2: 异步索引构建

📝 问题描述: 写操作同步更新所有索引，增加事务延迟。

✅ 优化方案: 主表同步写入，索引异步构建，最终一致性。

🎯 对应方法论: Deferring + Relaxation — 延迟 + 放宽一致性

// 创建延迟索引
CREATE INDEX idx_async ON orders(order_date) 
    WITH ASYNC_BUILD = true;  // 异步构建

// 写事务只更新主表，索引后台构建
INSERT INTO orders VALUES (...);  // 低延迟写入
// 索引在后台异步更新

优化点 3.3: Paxos 日志流水线

📝 问题描述: 传统 2PC 需要 4 次网络往返，协调者阻塞等待。

✅ 优化方案: 优化为 2 次 Paxos 同步，协调者状态不持久化，动态重建。

🎯 对应方法论: Task Reordering + Relaxation

7.1 LSM-Tree Flush 路径优化

// src/storage/lsmtable/memtable.cpp
// 当前: 单表 Flush
void MemTable::Flush() {
    // ... 同步 Flush 逻辑，阻塞写入
    sstable_writer_.Write(this);
    log_.Sync();  // 同步刷盘
}

// 优化: 批量异步 Flush
// 方法论: Batching + Deferring

class BatchFlushManager {
    struct FlushBatch {
        vector<MemTable*> tables;
        size_t total_size;
        Timestamp deadline;
    };
    
    void ScheduleFlush(MemTable* table) {
        std::lock_guard<mutex> lock(mutex_);
        pending_tables_.push_back(table);
        
        // 批量条件触发
        if (ShouldFlushBatch()) {
            // 提交后台线程异步处理
            bg_thread_.Submit(CreateBatch());
        }
    }
    
    bool ShouldFlushBatch() {
        return (TimeSinceLastFlush() > FLUSH_INTERVAL) ||
               (TotalPendingSize() > BATCH_SIZE_THRESHOLD);
    }
    
    void BackgroundFlush(FlushBatch batch) {
        // 低峰期执行，避免影响前台写入
        SStableWriter::WriteBatch(batch.tables);
        log_.AsyncSync();  // 异步刷盘
    }
};

// src/clog/paxos_log_writer.cpp
// 当前: 单条日志同步
Status PaxosLogWriter::Append(const LogEntry& entry) {
    return PaxosPropose(entry);  // 每次 RPC
}

// 优化: 批量组提交
// 方法论: Batching

class BatchPaxosWriter {
    struct LogBatch {
        vector<LogEntry> entries;
        vector<Promise<Status>> promises;
        Timestamp submit_time;
    };
    
    // 事务提交时入队，不立即同步
    Future<Status> AppendAsync(const LogEntry& entry) {
        Promise<Status> promise;
        auto future = promise.get_future();
        
        std::lock_guard<mutex> lock(batch_mutex_);
        current_batch_.entries.push_back(entry);
        current_batch_.promises.push_back(std::move(promise));
        
        // 触发条件检查
        if (current_batch_.entries.size() >= BATCH_SIZE ||
            TimeSinceFirstEntry() > BATCH_TIMEOUT_MS) {
            FlushBatch();
        }
        
        return future;
    }
    
    void FlushBatch() {
        // 单次 Paxos 提议整批日志
        auto batch = std::move(current_batch_);
        paxos_group_->ProposeBatch(batch.entries, [batch](Status s) {
            // 批量回调，通知所有等待事务
            for (auto& promise : batch.promises) {
                promise.set_value(s);
            }
        });
    }

7.4 负载自适应执行引擎

// src/sql/engine/adaptive_executor.cpp
// 方法论: Contextualization + Layering

class AdaptiveExecutor {
    struct WorkloadContext {
        WorkloadType type;           // OLTP / OLAP / MIXED
        double cpu_percent;          // CPU 使用率
        double memory_percent;       // 内存使用率
        double io_wait_percent;      // IO 等待
        uint64_t active_connections; // 活跃连接数
    };
    
    ExecutionStrategy SelectStrategy(WorkloadContext ctx) {
        // OLTP 场景优化
        if (ctx.type == WorkloadType::OLTP) {
            if (ctx.cpu_percent < 70 && ctx.memory_percent < 80) {
                // 资源充足: 激进缓存策略
                return Strategy::AGGRESSIVE_CACHE;
            } else {
                // 资源紧张: 减少缓存，避免 OOM
                return Strategy::MEMORY_CONSTRAINED;
            }
        }
        
        // OLAP 场景优化
        if (ctx.type == WorkloadType::OLAP) {
            if (ctx.io_wait_percent > 50) {
                // IO 瓶颈: 启用列存 + 预读取
                return Strategy::COLUMNAR_SCAN_WITH_PREFETCH;
            } else {
                // CPU 瓶颈: 向量化执行 + SIMD
                return Strategy::VECTORIZED_EXECUTION;
            }
        }
        
        // HTAP 混合场景: 租户隔离 + 资源组调度
        return Strategy::ISOLATED_RESOURCE_GROUPS;
    }
    
    // 动态线程池调整
    void AdjustThreadPool(WorkloadContext ctx) {
        if (ctx.type == WorkloadType::OLTP) {
            // OLTP: 大量短连接线程
            thread_pool_.SetConfig({
                .min_threads = 100,
                .max_threads = 500,
                .queue_size = 1000
            });
        } else {
            // OLAP: 少量长运行线程
            thread_pool_.SetConfig({
                .min_threads = 10,
                .max_threads = 50,
                .queue_size = 100
            });
        }
    }
};
