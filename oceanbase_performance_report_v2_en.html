📚 I. Theoretical Foundation: Sujin Park Optimization Framework

This report is based on the systematic optimization framework from the OSDI 2025 paper "Principles and Methodologies for Serial Performance Optimization" (Sujin Park, Mingyu Guan, Xiang Cheng, Taesoo Kim, Georgia Tech).

🎯 Core Idea:

Performance optimization is science, not magic. Through structured frameworks to identify bottlenecks and apply corresponding methodologies, performance can be systematically improved.

Three Optimization Principles

Principle Core Idea Application Scenarios
Task Removal Eliminate unnecessary work Identify and remove redundant computation, I/O, network transmission
Task Replacement Replace with more efficient implementation Algorithm optimization, architecture refactoring, hardware acceleration
Task Reordering Adjust execution order to reduce dependencies Asynchronization, pipelining optimization, deferred execution

Eight Actionable Methodologies

Batching

Combine multiple small operations → Reduce syscall, RPC count

Caching

Reuse previously computed results → Avoid repeated computation

Precomputing

Compute results needed in the future ahead of time → Eliminate runtime latency

Deferring

Defer non-urgent tasks → Decouple critical paths

Relaxation

Relax consistency/accuracy requirements → Reduce synchronization wait

Contextualization

Leverage context information for optimization → Load awareness, location awareness

Hardware Specialization

Optimize for specific hardware → Heterogeneous computing, RDMA, GPU

Layering

Layered abstraction to reduce overhead → Storage tiering, separation of concerns

🔷 III. Principle 1: Task Removal

💡 Core Idea: Eliminate unnecessary work

Identify redundant computation, duplicate requests, and invalid writes in the system, and remove them completely.

Optimization 1.1: Execution Plan Deduplication Cache

📝 Problem Description: Parameterized queries generate new plans each time. Even with identical query patterns, syntax parsing and cost estimation are repeated.

✅ Optimization Solution: Cache execution plans based on query templates. Queries with the same pattern directly reuse cached plans.

🎯 Corresponding Methodology: Caching — Reuse previously computed results

// Optimization configuration: Execution plan cache parameters
set global ob_plan_cache_percentage = 20;        // Cache memory ratio
set global ob_plan_cache_evict_high_percentage = 90;  // High watermark

// SQL Hint to use cache
SELECT /*+ USE_PLAN_CACHE(none) */ ...;          // Force no cache
SELECT /*+ USE_PLAN_CACHE(default) */ ...;       // Default caching policy

Optimization 1.2: Eliminate Repeated Hot Data Decompression

📝 Problem Description: When hot rows are read frequently, each read requires decompression from SSTable, causing significant duplicate computation.

✅ Optimization Solution: Introduce row-level cache layer. Hot data resides in memory to avoid repeated decompression.

🎯 Corresponding Methodology: Caching + Layering — Tiered caching strategy

// Row cache configuration
alter system set enable_row_cache = true;
alter system set row_cache_priority = 'HIGH';    // Hot rows cached with priority

// Specify cache policy during query
SELECT /*+ READ_CONSISTENCY(WEAK) */ * FROM hot_table WHERE id = 1;
// Weak consistency reads can hit row cache, avoiding SSTable access

Optimization 1.3: Data-Log Separation Storage

📝 Problem Description: Traditional Paxos requires data and logs to have the same replica count, resulting in high storage costs.

✅ Optimization Solution: Use 5 replicas for logs to ensure high availability, and 2-4 replicas for user data, removing redundant data replicas.

🎯 Corresponding Methodology: Layering — Storage tiering, separation of concerns

// Data-log separation configuration
ALTER SYSTEM SET data_copy_count = 2;      // User data 2 replicas
ALTER SYSTEM SET log_copy_count = 5;       // Logs 5 replicas

// Benefit: Storage cost reduced by 20-40% while maintaining same availability

🔷 IV. Principle 2: Task Replacement

💡 Core Idea: Replace with more efficient implementation

📝 Problem Description: Fixed execution strategies cannot adapt to OLTP/OLAP mixed workload fluctuations, resulting in severe resource contention.

✅ Optimization Solution: Dynamically select execution strategies based on workload context. Use row store for OLTP and column store for OLAP.

🔷 V. Principle 3: Task Reordering

💡 Core Idea: Adjust execution order to reduce dependencies/latency

Convert blocking operations to asynchronous, reorder task execution, and decouple critical paths.

Optimization 3.1: Asynchronous Compaction Scheduling

📝 Problem Description: Synchronous Compaction blocks writes, affecting transaction latency during peak hours.

✅ Optimization Solution: Defer Compaction during peak hours, batch execution during off-peak hours, decoupling critical paths.

🎯 Corresponding Methodology: Deferring — Defer non-urgent tasks

// Original order: Synchronous processing (blocking)
Write data → MemTable full → Immediate Flush → SSTable → Response to client
                    ↑________________________________________↓
                              (Blocks 5-10ms)

// Optimized: Asynchronous processing (non-blocking)
Write data → MemTable full → Enqueue → Immediate response
                        ↓
                     Background async Flush (executed during off-peak)

// Configuration parameters
ALTER SYSTEM SET minor_merge_concurrency = 4;      // Limit concurrency
ALTER SYSTEM SET minor_freeze_times = 10;          // Trigger threshold

Optimization 3.2: Asynchronous Index Building

📝 Problem Description: Write operations synchronously update all indexes, increasing transaction latency.

✅ Optimization Solution: Primary table synchronous write, asynchronous index building, eventual consistency.

🎯 Corresponding Methodology: Deferring + Relaxation — Deferral + Relaxing consistency

// Create deferred index
CREATE INDEX idx_async ON orders(order_date) 
    WITH ASYNC_BUILD = true;  // Asynchronous building

// Write transaction only updates primary table, indexes built in background

// Optimization: Batch async Flush
// Methodology: Batching + Deferring

class BatchFlushManager {
    struct FlushBatch {
        vector<MemTable*> tables;
        size_t total_size;
        Timestamp deadline;
    };
    
    void ScheduleFlush(MemTable* table) {
        std::lock_guard<mutex> lock(mutex_);
        pending_tables_.push_back(table);
        
        // Batch condition trigger
        if (ShouldFlushBatch()) {
            // Submit to background thread for async processing
            bg_thread_.Submit(CreateBatch());
        }
    }
    
    bool ShouldFlushBatch() {
        return (TimeSinceLastFlush() > FLUSH_INTERVAL) ||
               (TotalPendingSize() > BATCH_SIZE_THRESHOLD);
    }
    
    void BackgroundFlush(FlushBatch batch) {
        // Execute during off-peak hours to avoid impacting foreground writes
        SStableWriter::WriteBatch(batch.tables);
        log_.AsyncSync();  // Async disk flush
    }
};

7.3 Paxos Log Batch Replication Optimization

// src/clog/paxos_log_writer.cpp
// Current: Single log synchronous
Status PaxosLogWriter::Append(const LogEntry& entry) {
    return PaxosPropose(entry);  // Each RPC
}

// Optimization: Batch group commit
// Methodology: Batching

class BatchPaxosWriter {
    struct LogBatch {
        vector<LogEntry> entries;
        vector<Promise<Status>> promises;
        Timestamp submit_time;
    };
    
    // Enqueue during transaction commit, don't sync immediately
    Future<Status> AppendAsync(const LogEntry& entry) {
        Promise<Status> promise;
        auto future = promise.get_future();
        
        std::lock_guard<mutex> lock(batch_mutex_);
        current_batch_.entries.push_back(entry);
        current_batch_.promises.push_back(std::move(promise));
        
        // Trigger condition check
        if (current_batch_.entries.size() >= BATCH_SIZE ||
            TimeSinceFirstEntry() > BATCH_TIMEOUT_MS) {
            FlushBatch();
        }
        
        return future;
    }
    
    void FlushBatch() {
        // Single Paxos proposal for entire batch of logs
        auto batch = std::move(current_batch_);
        paxos_group_->ProposeBatch(batch.entries, [batch](Status s) {
            // Batch callback, notify all waiting transactions
            for (auto& promise : batch.promises) {
                promise.set_value(s);
            }
        });
    }
    
    // Benefit: N RPCs → 1 batch RPC, reduced latency, improved throughput

7.4 Load-Adaptive Execution Engine

// src/sql/engine/adaptive_executor.cpp
// Methodology: Contextualization + Layering

class AdaptiveExecutor {
    struct WorkloadContext {
        WorkloadType type;           // OLTP / OLAP / MIXED
        double cpu_percent;          // CPU usage
        double memory_percent;       // Memory usage
        double io_wait_percent;      // IO wait
        uint64_t active_connections; // Active connections
    };
    
    ExecutionStrategy SelectStrategy(WorkloadContext ctx) {
        // OLTP scenario optimization
        if (ctx.type == WorkloadType::OLTP) {
            if (ctx.cpu_percent < 70 && ctx.memory_percent < 80) {
                // Sufficient resources: Aggressive caching policy
                return Strategy::AGGRESSIVE_CACHE;
            } else {
                // Resource constrained: Reduce cache to avoid OOM
                return Strategy::MEMORY_CONSTRAINED;
            }
        }
        
        // OLAP scenario optimization
        if (ctx.type == WorkloadType::OLAP) {
            if (ctx.io_wait_percent > 50) {
                // IO bottleneck: Enable columnar storage + prefetch
                return Strategy::COLUMNAR_SCAN_WITH_PREFETCH;
            } else {
                // CPU bottleneck: Vectorized execution + SIMD
                return Strategy::VECTORIZED_EXECUTION;
            }
        }
        
        // HTAP mixed scenario: Tenant isolation + resource group scheduling
        return Strategy::ISOLATED_RESOURCE_GROUPS;
    }
    
    // Dynamic thread pool adjustment
    void AdjustThreadPool(WorkloadContext ctx) {
        if (ctx.type == WorkloadType::OLTP) {
            // OLTP: Large number of short-connection threads
            thread_pool_.SetConfig({
                .min_threads = 100,
                .max_threads = 500,
                .queue_size = 1000
            });
        } else {
            // OLAP: Small number of long-running threads
            thread_pool_.SetConfig({
                .min_threads = 10,
                .max_threads = 50,
                .queue_size = 100
            });
        }
    }
};
