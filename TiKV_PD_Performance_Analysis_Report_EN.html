📚 Theoretical Foundation: Sujin Park Optimization Framework

This report is based on the OSDI 2025 paper "Principles and Methodologies for Serial Performance Optimization" by Sujin Park, Mingyu Guan, Xiang Cheng, and Taesoo Kim from Georgia Tech.

🎯 Core Idea: Performance optimization is science, not magic. Through structured frameworks, we identify bottlenecks, apply corresponding methodologies, and systematically improve performance.

Three Core Optimization Principles

Principle Core Idea Application Scenarios
Task Removal Eliminate unnecessary work Identify and remove redundant computation, I/O, RPC
Task Replacement Replace with more efficient implementation Algorithm optimization, architecture refactoring
Task Reordering Adjust execution order to reduce dependencies Async processing, pipeline optimization

Eight Actionable Methodologies

Batching — Combine multiple small operations → Reduce syscalls, RPC count
Caching — Reuse previously computed results → Avoid redundant computation
Precomputing — Compute future-needed results ahead of time → Eliminate runtime latency
Deferring — Delay non-urgent tasks → Decouple critical paths
Relaxation — Relax consistency/accuracy requirements → Reduce synchronization wait
Contextualization — Leverage contextual information for optimization → Location-aware, load-aware
Hardware Specialization — Optimize for specific hardware → Heterogeneous computing, microservices split
Layering — Layered abstraction reduces overhead → Storage tiering, separation of concerns

🔷 Principle 1: Task Removal

💡 Core Idea: Eliminate Unnecessary Work

Identify redundant computation, duplicate requests, and invalid writes in the system, and remove them entirely.

Optimization 1.1: TSO Batch Prefetching to Reduce RPC Calls

📝 Problem Description

Each transaction requests TSO multiple times, generating significant RPC overhead.

✅ Optimization Solution

Batch prefetch TSO on TiKV side, cache locally, and significantly reduce RPC count.

🎯 Corresponding Methodologies

Batching + Caching

// Optimization example: TiKV-side TSO prefetch configuration
[causal-ts]
alloc-ahead-buffer = "6s"        # Pre-allocate 6 seconds of TSO
renew-batch-max-size = 65536     # Maximum batch size
renew-batch-min-size = 2048      # Minimum batch size

// Principle: From N RPCs → 1 batched RPC

Optimization 1.2: Remove Unnecessary etcd Writes

📝 Problem Description

Heartbeat data is fully written to etcd, generating significant disk I/O and impacting performance.

✅ Optimization Solution

Distinguish between volatile data and persistent data. Store volatile data in memory, persistent data in etcd.

🎯 Corresponding Methodology

Layering — Tiered storage strategy

// Data tiering strategy
Volatile data (Region real-time state) → Memory storage (fast path)
Persistent data (Region topology)     → etcd storage (slow path)

🔷 Principle 2: Task Replacement

💡 Core Idea: Replace with More Efficient Implementation

Preserve functional semantics while replacing with better algorithms, architectures, or data structures.

Optimization 2.1: Microservices Architecture Decoupling (PD v8.0+)

📝 Problem Description

PD monolithic architecture: TSO/scheduling/routing affect each other; one module failure impacts the entire system.

✅ Optimization Solution

Split into independent microservices, each module deploys and scales independently.

🎯 Corresponding Methodology

Hardware Specialization — Specialized deployment for different workloads

Optimization 2.2: Local TSO Replacing Global TSO (Cross-DC Scenario)

📝 Problem Description

In cross-DC scenarios, all TSO requests route to PD leader, incurring 20-50ms latency.

✅ Optimization Solution

Introduce Local TSO allocator. Local transactions use local timestamps; only cross-DC transactions need Global TSO.

🎯 Corresponding Methodology

Contextualization — Leverage geographical context for optimization

// Local TSO architecture
DC-1: Local TSO Allocator ← Local transactions (low latency < 5ms)
DC-2: Local TSO Allocator ← Local transactions (low latency < 5ms)
       ↓
Global TSO Allocator ← Cross-DC transactions (sync overhead)

🔷 Principle 3: Task Reordering

💡 Core Idea: Adjust Execution Order to Reduce Dependencies/Latency

Convert blocking operations to async, reorder task execution, and decouple critical paths.

Optimization 3.1: Asynchronous Heartbeat Processing

📝 Problem Description

Synchronous heartbeat processing blocks critical paths; TiKV faces timeout risks waiting for responses.

✅ Optimization Solution

Respond immediately upon receiving heartbeat, process state updates asynchronously.

🎯 Corresponding Methodology

Deferring — Delay non-urgent tasks

// Original order: Synchronous processing
Receive heartbeat → Update state → Persist storage → Respond heartbeat
   ↑___________________________________↓
              (Block 5-10ms)

// Optimized: Asynchronous processing
Receive heartbeat → Enqueue → Respond immediately
             ↓
          Async state update

Optimization 3.2: TSO Physical Time Advancement Optimization

📝 Problem Description

TSO physical time checks every 50ms; short transactions may wait unnecessarily.

✅ Optimization Solution

Dynamically adjust check interval; update physical timestamp at higher frequency.

🎯 Corresponding Methodology

Relaxation — Trade strictness for performance

4.1 TSO Allocation Path Optimization

// pkg/tso/allocator.go
// Current: Single batch allocation
func (a *Allocator) GenerateTSO(count uint32) (pdpb.Timestamp, error) {
    // ... synchronous allocation logic, each request goes through network
}

// Optimized: Batch prefetching + local caching
// Methodologies: Batching + Caching

type TSOCache struct {
    buffer []pdpb.Timestamp
    index  int
    mu     sync.Mutex
}

func (c *TSOCache) Get(count uint32) (pdpb.Timestamp, error) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    if c.index+int(count) >= len(c.buffer) {
        // Batch fetching - Batching methodology
        c.refill()  // Fetch large batch of TSO at once
    }
    
    // Return from cache - Caching methodology
    ts := c.buffer[c.index]
    c.index += int(count)
    return ts, nil
}

// Optimized: Asynchronous batch processing
// Methodologies: Batching + Deferring

func (c *RaftCluster) processRegionHeartbeat(region *core.RegionInfo) error {
    // 1. Immediately update memory state (critical path, must be fast)
    c.basicCluster.PreCheckPutRegion(region)
    
    // 2. Async batch persistence (non-critical path, process later)
    c.heartbeatQueue.Push(region)
    
    return nil  // Return immediately, no blocking
}
